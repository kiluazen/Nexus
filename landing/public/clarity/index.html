Nexus · field guide

How AI-platform widgets actually work

The protocol, the two platforms that run it differently, and every single thing that bit us — in one page you can read on a phone.

Written 14 July 2026 · from live tests on ChatGPT + Claude web/iOS + the official docs of both · updated as things change (and they will)

00 · The 60-second version

TL;DR

  • There is now one open protocol for widgets inside AI chats: MCP Apps, an official MCP extension since 26 Jan 2026. Your server ships HTML; the platform runs it in a sandboxed iframe; they talk JSON-RPC over postMessage.
  • ChatGPT and Claude both run it. ChatGPT keeps extra openai/* features on top (not deprecated, just optional). Claude is closest to the pure standard.
  • The Expo-like thing you asked for exists and is official: the @modelcontextprotocol/ext-apps SDK. Nexus now uses its official App client on both hosts; one shared protocol bridge, no hand-rolled fallback.
  • Tool sequencing is a model decision, not an MCP-server primitive. Nexus never distorts read/write semantics to work around a host's widget-placement bug: correctness and truthful mutation results come first.
  • Nexus today: the shared MCP Apps implementation has rendered on ChatGPT web/iOS and Claude web/iOS. Each deployment keeps only its host-specific policy and optional enhancement layer.
  • Nothing you felt was imaginary: the platforms genuinely rewrote their format mid-flight (Oct 2025 → Jan 2026), the docs lag the hosts, and dev mode has its own bugs. The fix is testing on real hosts + a few hard rules (section 08).
01 · How we got here

The story: three formats in nine months

Your anger is justified. Anyone who built a ChatGPT app in late 2025 has now lived through two format migrations. Here's the actual sequence:

  • EARLY 2025mcp-ui — a community project (Ido Salomon, Liad Yosef) proves you can serve UI over MCP. No platform support yet.
  • OCT 2025ChatGPT Apps SDK launches — OpenAI's own, proprietary take: window.openai global injected into your iframe, openai/outputTemplate to link tool→widget, mimeType text/html+skybridge. This is "the old format".
  • NOV 2025SEP-1865 proposed — the mcp-ui authors + OpenAI + Anthropic take it to the MCP standards process.
  • 26 JAN 2026MCP Apps becomes an official MCP extension, with the @modelcontextprotocol/ext-apps SDK. Claude (web+desktop) supports it day one; ChatGPT follows within days; VS Code, Goose, JetBrains after.
  • SPRING 2026ChatGPT adopts the standard keys (_meta.ui.*, text/html;profile=mcp-app) and demotes its own keys to "compatibility aliases + ChatGPT-only extensions". Official line: lead with the standard, no plans to deprecate openai/*.
  • JUL 2026Nexus converges on the official standard client — one ext-apps App bridge on both hosts, separate host deployments, shared tools/data/widget core, and successful renders across web and iOS.

So when the docs and skills files contradict each other: they were written in different eras. Anything mentioning text/html+skybridge or only window.openai is era-2 material. Anything with _meta.ui.resourceUri and ui/initialize is current.

02 · The mental model

The protocol on one page

Strip away every brand name and it's this: your MCP server ships a self-contained HTML file; the platform runs it in a sandboxed iframe next to the chat; the iframe and the platform exchange JSON-RPC messages.

Server side — three declarations

  1. A UI resource. An MCP resource whose URI uses the ui:// scheme (e.g. ui://widget/nexus-today-v15.html) and whose mimeType is text/html;profile=mcp-app. Its text is your entire widget — HTML + all JS inlined, because the sandbox blocks most external fetches.
  2. A pointer on the tool. The tool that should render the widget carries _meta.ui.resourceUri naming that resource. Tool fires → platform fetches the resource → mounts the iframe.
  3. Policy on the resource. _meta.ui.csp (which hosts the iframe may fetch/connect to), _meta.ui.domain (a stable serving origin, each platform has its own rules), _meta.ui.prefersBorder (chrome hint).

Client side — the handshake

Two details that matter in practice: the initialize params must be {protocolVersion, capabilities, clientInfo} (not made-up field names — hosts may validate), and ui/resource-teardown is a request, so a widget that never answers host requests is technically hung. The official App class handles all of this; a hand-rolled bridge has to remember to.

05 · The Expo question

"Is there no mature build-once-run-everywhere?"

There is — it's just young. And unusually, the official SDK is the Expo, not a third party.

@modelcontextprotocol/ext-apps is explicitly designed for cross-host:

  • registerAppTool() / registerAppResource() on the server — you declare once, it emits the platform-specific metadata for both hosts.
  • App.connect() in the widget — auto-detects whether it's inside ChatGPT or Claude and picks the right transport.
  • Handles the whole request surface you'd forget by hand: ping, ui/resource-teardown, auto-resize, correct initialize params.
  • Examples in vanilla JS, React, Vue, Svelte + an official migration guide from the old OpenAI Apps SDK + a Claude Code skills plugin (/plugin marketplace add modelcontextprotocol/ext-apps).

Historical caveat from our own week: one early inlined official-runtime build crashed in ChatGPT and a tiny direct shim helped isolate the failure. The final architecture did not preserve that fork: the build issue was removed and production now uses the official client on both targets.

And the Expo analogy has the same fine print as actual Expo: the shared layer covers the protocol, but each store keeps its own rules — ChatGPT's review process and openai/* extras, Claude's domain hash and connector flow. You can share the app; you can't share the paperwork.

06 · Case files

Everything that bit us, with verdicts

CLOSED

"Failed to fetch template" on ChatGPT

The widget resource still declared the era-2 mimeType text/html+skybridge. ChatGPT's client now asks for text/html;profile=mcp-app and aborts on mismatch (confirmed by live-tailing prod).

Rule: the mimeType is a handshake, not a label.

CLOSED · historical

"Error loading app — Runtime error" on ChatGPT, every turn

An early bundled build of the official client crashed, so Nexus temporarily used a tiny direct implementation. That was not the final architecture: production now runs the official ext-apps App class on both hosts.

Rule: isolate the bundle/build failure; do not turn a temporary shim into architecture.

CLOSED

Claude refused to render: "Invalid ui.domain format"

Claude requires ui.domain to be its hash form. For our URL: 9b68940b…a79.claudemcpcontent.com — the hash covers the exact connector URL string, trailing slash included.

Rule: compute the hash from the URL in Settings→Connectors, character for character.

CLOSED

Widget renders but shows a stale version

Hosts cache the widget resource by URI. Same URI + new HTML = old widget. ChatGPT sometimes also keeps a stale tool→widget binding until the app is deleted and recreated (our Mixtome/Coffeetable lesson).

Rule: bump the ui://…vN version on every behavioral change. If ChatGPT still won't update: delete + recreate the app, refresh won't do it.

CORRECTED

Multi-tool turns were mistaken for an application constraint

Claude can mount the widget after a history→log sequence, but may place it inside collapsed tool/reasoning UI. ChatGPT iOS has separately dropped cards on some multi-call turns. Those are host presentation behaviors, not permission to skip a requested mutation or claim success without a successful tool result.

Rule: never change data semantics to optimize card placement. Read when correctness requires it; never fabricate mutation success.

CLOSED

Claude iOS once painted the shell without data

The official App handshake, connect timeout, default paint, and correct Claude domain are now deployed; the hydrated card subsequently rendered on Claude iOS.

Rule: implement the complete lifecycle and verify on the actual mobile host.

OPEN · latent

Landmines we know about and haven't stepped on yet

Claude's ~150K-char result limit (widget gets a file pointer, never hydrates) — Nexus history payloads should stay paginated. ChatGPT refetches every resource every turn — keep the widget document lean. Claude mounts a new live iframe per call — old Nexus cards in a long chat are all still running; the supersession pattern exists when we care. iOS omits the Referer header — allowlist by Origin if the widget ever fetches cross-origin.

Rule: reread this section before each new feature, not after it breaks.

  • Mutation retries and old mounted cards are handled in the shared data layer with durable mutation identities and optimistic state versions.

  • Mocks validate protocol contracts; the real-host smoke matrix remains mandatory before each release.
08 · Doctrine

Rules to build by

  1. The standard is the base; extensions are per-store paperwork. Declare with ui.*; add openai/* only for ChatGPT-only features you actually want. That's not a fallback — there's nothing it falls back from.
  2. Correctness outranks card placement. Tool sequencing belongs to the model and host. Read state when needed, execute requested writes, and never claim success without a successful mutation result.
  3. Bump the ui://…vN URI on every behavioral change. Caches are ruthless; version numbers are free.
  4. Fail visibly. Timeout the handshake, paint a default state, show one quiet line when data never arrives. A blank card on someone's phone is undebuggable by definition.
  5. Keep payloads lean. Claude hard-stops hydration past ~150K chars; ChatGPT re-downloads your resources every turn.
  6. Trust only real hosts. Mocks prove your code is self-consistent, never that it works. Smoke-test on ChatGPT web, ChatGPT iOS, Claude web, Claude iOS before calling anything done.
  7. When mobile misbehaves, attach an inspector, don't guess. A host bug is evidence to report, not a constraint to encode into application semantics.
  8. Deploy → verify → commit → push. Same hour. Prod running uncommitted code is how you lose a working state you can't rebuild.
  9. Reread the docs monthly. This ecosystem shipped three formats in nine months. Assume this page has a half-life; check the sources below when something smells stale.

The frustration-saving heuristic: if your widget worked yesterday and you deployed nothing — it's them. If it works on one surface (web) and not another (mobile, dev mode, Claude Code) with identical code — it's almost certainly them. Your code runs in four host runtimes owned by two companies shipping weekly; an unchanged widget breaking is expected weather, not your failure.

What makes it diagnosable in minutes instead of days: fail visibly (rule 4), test surfaces separately (rule 6), and keep this page's receipts handy so you can say "known issue, moving on".
